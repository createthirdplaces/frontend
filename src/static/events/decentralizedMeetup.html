Decentralized meetup alternative

Createthirdplaces.com is working on open source software for people to find in person events as a way of promoting third places.

Here are the guidelines that will be followed for any changes See the vision page for more details.
  • Support in person interaction
  • All content is human generated
  • No personalized recommendations
  • Non-commercial
  • Decentralization

Details about the decentralized architecture

The general idea behind the decentralized architecture will involve having 2 types of sites. Local websites for a specific activities and aggregator websites. Local websites will be independently created and run while sharing data through an API. Aggregator websites will read data from multiple local sites and allow users to search through multiple sites at the same time. Aggregator websites can also be aggregators for other websites.

For example, there can be a local websites in DC for chess events and a website for playing resource management games such as Catan. A board game aggregator website could then read information from the chess website and Catan website, and allow users to search for events related to both activities. The board game aggregator website could then have data read by a DC area aggregator website that combines information from other types of websites such as DC sporting event website.

One way of efficiently implementing a decentralized architecture would be to make sure websites fulfill the following guidelines

  • When a group is created on a local website, the information is saved to a database of events for the website with a UUID as the primary key. The events would then be accessible through a public API.
  • Aggregator websites save the UUIDs of all groups in their aggregation and include references to the API endpoint of the original site location. This information will be available through an API endpoint for other aggregator websites.
  • When an aggregator website requests information about a specific group, it will retrieve the UUID and local website URL from its database. This can then be used to perform a query on the original website through multiple aggregation layers without significant performance degradation.
  • All aggregator websites will need to register with the websites they are aggregating from. After registering, the aggregator will retrieve all the groups from the website they registered with. When a group is created or deleted, it will look at the websites that registered as aggregators, and then call an API endpoint for the aggregator to make sure it has information about the group.

Challenges that this architecture will address

  • Efficiently reading updated data from other websites. If an aggregator website has to check other websites to get information and those websites are also aggregators, then it will be important to optimize performance. Also, circular dependencies will also need to be handled.
  • Handling authentication and user permissions. One possible way to address the complexity would be for people to have the ability to join group or modify groups they created through aggregator websites. This would help with usability. However, it will add complexity in the short term and could make things too centralized.
  • Handling issues with too many requests from a location

Reasons why a decentralized network of third websites is a good idea
  • It is difficult for a general purpose website to meet the interests of specific hobbies or communities. People have a variety of different interests that will be complex to effectively represent on one website. Also, it is impossible for the developers of one site to have knowledge about every single community and hobby.
  • Smaller websites can have content related to the interests of their audience without affecting how others see content they want to see. As an example, dmvboardgames.com, a website promoting in person board games in the DC area has a page for local board game designers.
  • Local community organizers can submit event information to a large aggregator website without being dependent on them. If the large website does something similar to Meetup.com and starts charging $300 per year to host groups while downgrading their UI, the owners can move to a different aggregator or not use one.
  • The ability to connect with Fediverse websites through ActivityPub.

Issues with other third place websites

In the past, there were a variety of sites that people could use to promote events in third places. Unfortunately many of these sites have declined in quality or prioritize keeping people online. The website Meetup.com was a notable exception as it prioritized in person events. However, the quality of Meetup has declined, and it is no longer a good option for hosting events. See this page for more details.


Other platforms are attempting to compete with Meetup such as Luma and Partiful. However, they follow the same VC funded for profit business model used by Meetup, and are on the path to enshittification This includes one former competitor Eventbrite, which has been purchased by the company Bending Spoons . This is the same company that owns Meetup.com and has been responsible for its latest problems.
