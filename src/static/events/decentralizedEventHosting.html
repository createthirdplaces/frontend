Decentralized event hosting

Create Third Places is working on open source software for people to find in person events as a way of promoting third places. It will be designed as a decentralized open source alternative to Meetup.com.

Guidelines that will be followed for any changes

  • Support in person interaction
  • All content is human generated
  • No personalized recommendations
  • No commercial monetization
  • Promote decentralization

See the vision page for more details.

Overview of decentralized architecture

The decentralized architecture includes 2 types of websites:

  1. Event websites:Websites for a specific type or activity or area They will share event and group data through an API.
  2. Aggregator websites: They will read data from local websites and allow users to search their data.

Websites can be both aggregator websites and event websites.

More details about decentralized architecture

One example would a website for chess events in DC and another website for resource management board games such as Catan. An aggregator website could then read information from the chess website and Catan website, and allow users to search for events related to both games. The board game aggregator website could then have its data read by a DC area aggregator website that combines information from other types of websites such as a DC sports event website.

While this architecture is decentralized, it might be desirable for an organization to have an implementation of this under centralized control. One example would be a local parks agency managing events for multiple parks. In this case, decentralization would come from the fact that managers of individual parks can have event sites, while sharing the events with an aggregator that can be used to search for events across multiple parks.

A similar architecture has seen success in settings where centralized control outside of the computer systems is necessary. For example The Create Third Places decentralized architecture is similar to the Bell-LaPadula security model with discretionary access control which was used by the US Department of Defense.

To summarize, the Create Third Places architecture and any other decentralized architecture has an effective decentralization level based on the centralization level of the outside entities controlling the individual parts. Create Third Places, blockchain, or any other decentralized architecture is not capable of forcing decentralization on a system that an outside entity wants to centralize.

Websites using the decentralized architecture will follow these guidelines
  • When a group is created on a local website, the information is saved to a database of events for the website with a UUID as the primary key. The events would then be accessible through a public API.
  • Aggregator websites save the UUIDs of all groups in their aggregation and include references to the API endpoint of the original site location. This information would then be available through an API endpoint for other aggregator websites.
  • When an aggregator website requests information about a specific group, it will retrieve the UUID and local website URL from its database. This information can then be used to perform a query on the local website through multiple aggregation layers without significant performance degradation. An aggregator website will also be able to request a list of events.
  • All aggregator websites will need to register with the websites they are aggregating from. After registering, the aggregator will retrieve all the groups from the website they registered with. When a group is created or deleted, it will look at the websites that registered as aggregators, and then call an API endpoint for the aggregator to make sure it has information about the group.
  • To prevent loops, each aggregator will have a level specified by an integer ranging from 0 to 2,147,483,647. A website will not be able to aggregate data from a higher level. Local event websites should have a level of 0 to have their data to be read by aggregators. Any website that will not support aggregators should have a level of 2,147,483,647.
Challenges that this architecture will address
  • Efficiently reading updated data from other websites. If an aggregator website has to check other websites to get information and those websites are also aggregators, then it will be important to optimize performance.
  • Handling authentication and user permissions. One possible way to address the complexity would be for people to have the ability to join group or modify groups they created through aggregator websites. This would help with usability. However, it would add complexity in the short term and could make things too centralized.
  • Handling issues with too many requests from a location.
  • Resiliency against being compromised by a single point of failure. If an aggregator website crashes or is affected by hackers, a new aggregator website can be created with the data stored on local websites. Also, the decentralized architecture means that malicious actors do not have a single target they can focus on. The need for aggregators to have permission from local sites to retrieve data means that local sites can also decide to cancel permissions to block aggregators.

Reasons why a decentralized network of third websites is a good idea
  • It is difficult for a general purpose website to meet the interests of specific hobbies or communities. People have a variety of different interests that will be complex to effectively represent on one website. Also, it is impossible for the developers of one site to have knowledge about every single community and hobby.
  • Smaller websites can have content related to the interests of their audience without affecting how others see content they are interested in. As an example, dmvboardgames.com, a website promoting in person board games in the DC area has a page for local board game designers.
  • Local community organizers can submit event information to a large aggregator website without being dependent on them. If the large website does something similar to Meetup.com and starts charging $300 per year to host groups while downgrading their UI, the owners can move to a different aggregator or not use one.
  • The ability to connect with Fediverse websites through ActivityPub.
  • It is easier for individual communities to move to an alternative. If the quality of an aggregator website declines, users can switch to an alternative aggregator site without losing group and event data that is on a local website. Local websites owners can also remove access to an aggregator if they think that aggregator should no longer be used.

Issues with other websites used to promote third places

In the past, there were a variety of websites that people could use to promote events in third places. Unfortunately many of them have declined in quality or started to prioritize keeping people online. The website Meetup.com was a notable exception as it prioritized in person events. However, the quality of Meetup has declined, and it is no longer a good option for hosting events. See this page for more details.

Other platforms are attempting to compete with Meetup such as Luma and Partiful. However, they follow the same VC funded for profit business model used by Meetup, and are on the path to enshittification. This includes one former competitor Eventbrite, which has been purchased by the company Bending Spoons. This is the same company that owns Meetup.com and has been responsible for its latest problems.
