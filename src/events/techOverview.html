Decentralized event hosting tech overview

Websites using this architecture can have one or both of the following types

  1. Group websites: Websites that share group data through an API.
  2. Aggregator websites: They will read data from local websites and allow users to search their data.

Websites can be both aggregator websites and events websites. Also, if an aggregator website is also an event website, the website APIs could return a subset of events. For example, an aggregator website for weekend events in a city, could read data from a group website to get a list of groups in a city, and filter for events that are on Friday evening, Saturday, or Sunday, and then return weekend events through their API.

Websites using the decentralized architecture will follow these guidelines
  • When a group is created on a group website, the information is saved to a database of groups for the website with a UUID as the primary key. Events and groups would then be accessible through a public API.
  • Aggregator websites save the UUIDs of all groups in their aggregation and include references to the API endpoint of the original site location. This information would then be available through an API endpoint for other aggregator websites.
  • When an aggregator website requests information about a specific group, it will retrieve the UUID and local website URL from its database. This information can then be used to perform a query on the group website through multiple aggregation layers without significant performance degradation. An aggregator website will also be able to request a list of events.
  • All aggregator websites will need to register with the websites they are aggregating from. After registering, the aggregator will retrieve all the groups from the website they registered with. When a group is created or deleted, it will look at the websites that registered as aggregators, and then call an API endpoint for the aggregator to make sure it has information about the group.
  • Aggregator websites will store a list of all group websites being used to prevent circular dependencies being caused by a registration attempt.
Challenges that this architecture will address
  • Efficiently reading updated data from other websites. If an aggregator website has to check other websites to get information and those websites are also aggregators, then it will be important to optimize performance.
  • Handling authentication and user permissions. One possible way to address the complexity would be for people to have the ability to join group or modify groups they created through aggregator websites. This would help with usability. However, it would add complexity in the short term and could make things too centralized.
  • Handling issues with too many requests from a location.
  • Resiliency against being compromised by a single point of failure. If an aggregator website crashes or is affected by hackers, a new aggregator website can be created with the data stored on group websites. Also, the decentralized architecture means that malicious actors do not have a single target they can focus on. The need for aggregators to have permission from local sites to retrieve data means that local sites can also decide to cancel permissions to block aggregators.
